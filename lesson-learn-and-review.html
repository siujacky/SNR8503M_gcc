1. Executive summary

This project ported a vendor-locked Keil-µVision BLDC motor-control firmware (SNR8503M / LKS32MC03x, ARM Cortex-M0) to a modular GCC-buildable codebase. Across roughly two days of work the firmware grew from a single fixed-feature Hall build into a 7-profile system with bipropellant binary+ASCII protocol, optional sensorless mode, runtime-tunable PID, and a Flash-backed configuration block — all selectable at compile time via a single PROFILE= flag.

The journey produced one significant near-miss (the startup .data-copy bug was silently corrupting initialised globals across every GCC build until the second-pass code review caught it), several non-obvious vendor-SDK quirks (closed libs with symbol clashes, STM32-flavoured dead code, undocumented register sequences), and a working multi-profile firmware that builds for 7 distinct hardware/feature combinations.

Headline numbers: 37 of 38 review findings fixed (97%)  ·  0 HIGH-severity issues remaining  ·  ~25 KB Flash, ~12 KB RAM for the default profile  ·  All under 32 KB MDK-Lite cap  ·  ~20 hours of iterative work

3. What worked well

ARMCC closed libs linked with GCC on first try. The two vendor-supplied .lib files (compiled with ARMCC V5) were expected to be a major blocker for the GCC migration. Adding -Wl,--no-warn-mismatch to the linker flags was sufficient — symbol-level compatibility held. This skipped the entire planned Phase 2 (stubs) and let the motor control path stay byte-identical.
Map-file inspection revealed vendor identity. The linker map showed lks32mc03x_nvr.o inside the SNANER-branded NVR lib. That single string identified SNR8503M as a re-marked LKS32MC03x from LINKO Semiconductor, which then unlocked: (a) finding the BSD-licensed LINKO platform_software repo, (b) understanding the closed-lib structure across the entire chip family, (c) knowing what to expect from the sensorless SDK.
Modular architecture preserved Hall-only byte equivalence. After Phase 10 refactored 12 files into #if MODULE_* guards, the hall_bipropellant_full profile produced the exact same hex as the pre-refactor build (28,380 → 28,380 bytes, then later 28,404 for explicit FIX-008 cleanup). This means existing Hall hardware was provably unaffected.
Code-review agent caught a HIGH-severity bug self-review missed. FIX-001 (the startup .data-copy loop incrementing both r0 and r3, doubling the stride after the first word) was a silent data-corruption bug. The author's own review missed it across multiple passes. A general-purpose code-review agent caught it within minutes by reading the startup asm critically. Single most valuable finding of the whole project.
Profile system pays off immediately. Adding 0xB4 firmware_features bitfield in iteration 2 was a 30-line change but immediately makes the firmware self-describing: a host reading 0xB4 sees exactly which modules are compiled. No more parsing the build log to know what's available.

4. Surprises & mistakes

Initial license claim about bipropellant-protocol was wrong. The author told the user bipropellant-protocol was GPL-2.0+ for several conversational turns. It is actually MIT. This led to multiple unnecessary discussions about GPL contamination of the closed vendor libs. The mistake was made on weak evidence (assumption from the related hoverboard-firmware repo's GPL-3) and was only corrected after explicitly fetching the LICENSE file. Lesson: always read the LICENSE file before claiming a license.
FIX-001 silently corrupted every GCC build for ~5 phases. The startup .data-copy loop bug was present from Phase 1.4 (initial startup conversion) through Phase 13. It would have invisibly corrupted any non-zero-initialised global — including the Hall commutation lookup tables. The firmware "worked" because most globals are zero-init, but any GCC build flashed during this period was suspect. Lesson: always verify startup correctness with a canary, especially when hand-translating between assembler dialects.
Phase 13 BLDC_COMP_Input_Only initial stub. The function was stubbed as a no-op for "build it works, motor doesn't" sensorless mode. This was honest but the impact was understated: without the real implementation, sensorless mode looks correct in the firmware but the motor would simply not commutate. The actual port (6-case switch writing SYS_AFE_REG1) was ~30 lines and should have been done in Phase 13 directly rather than left as a TODO.
Find_Motor_Rotor.c was STM32 dead code. The vendor sensorless SDK contained 512 lines of high-frequency-injection rotor-discovery code that referenced STM32 timer register names (TIM_CCMR1_OC1M_2 etc.) not defined for SNR8503M. The vendor never made this part work on this chip. The file was absorbed verbatim into the new tree before this was discovered. Lesson: don't trust vendor SDKs to be self-consistent; smoke-test compile each new file before integrating.
Pool[12] silently dropped 3 param registrations. The static PARAMSTAT pool[12] array in protocol_params_flash.c exactly matched the initial 12 codes (0x80-0x89, 0x90, 0xA0). When 3 motor-mode codes were added later (0xB0-0xB2 gated on MODULE_MOTOR_AUTODETECT), the pool silently overflowed — the third extra registration triggered the early-return guard, and 0xB0-0xB2 vanished without warning. Lesson: when adding to a fixed-size pool, audit consumers; or use a sentinel like _Static_assert.
Motor_step_tab_cw exists in BOTH SDKs with different sizes. Hall SDK defines uint8_t Motor_step_tab_cw[8] (Hall-state → step lookup); sensorless SDK defines uint8_t Motor_step_tab_cw[6] (step transition table). Same name, incompatible meaning. Resolved by (a) changing MC_Drive.h to use incomplete-type extern, and (b) opt-in rename macros in sensorless_adapter.h. Lesson: when merging two SDKs, scan globally-named arrays for size mismatches before integrating.

5. Patterns & techniques that worked

Module system via centralized config header

Single include/feature_config.h with 11 module flags + 5 compile-time #error validations. Every other file just adds #if MODULE_X / #endif around its body. Profile presets are generated by build.sh on demand. This kept refactoring boring and mechanical instead of architectural.

Vendor lib symbol clash via objcopy rename

Both SNR_BLDC_HALL_V1p0.lib and SNR_BLDC_SNLS_V1p0.lib export BLDC_init (and BLDC_1, BLDC_BUFF in .data). Build script runs arm-none-eabi-objcopy --redefine-sym BLDC_init=BLDC_init_snls ... to produce a renamed copy of the sensorless lib. Both libs then coexist cleanly. This trick saved a reverse-engineering project.

Adapter struct for incompatible globals

Sensorless SDK uses Motor_1M1st with its own struct layout (different from Hall's Motor). Rather than refactor either SDK, an adapter (include/sensorless_adapter.h + src/sensorless_adapter.c) provides a parallel global only when sensorless mode is compiled. Cost: ~150 bytes RAM; benefit: zero changes to vendor source.

Boot-time canary for startup regression

After FIX-001 was found, src/startup_canary.c defines volatile uint32_t __data_canary[4] = {0xCAFE000{1,2,3,4}} and startup_canary_check() verifies them at main() entry. If the startup ASM ever regresses to a buggy copy loop, this catches it. Cost: 16 bytes Flash + 16 bytes RAM + 4 word comparisons at boot.

Iterative review cycles

Pattern: write code, self-review, get external review, write fix-up-plan.json with findings, fix all, re-review for new issues introduced by the fixes, produce fix-up-plan-2.json, fix again. Two iterations took the project from "all 7 profiles build" to "0 HIGH-severity issues remaining." The second iteration was much shorter (13 findings vs 25) — diminishing returns confirm the approach converges.

7. Recommendations for future work

First action: bench-test on real hardware. The single most important next step. Flash build/snr8503m.hex (currently hall_bipropellant_full). Verify motor spins. Read parameters 0x08 (voltage/current), 0x04 (position), 0xB4 (features bitfield). The startup canary check + features bitfield read are quick sanity checks that diagnose 80% of potential issues in under a minute.
Sensorless verification requires a no-Hall test board. The sensorless code path is structurally complete but never run on hardware. Flash unified_full, disconnect Hall sensors. Verify 0xB0 reads MODE_SENSORLESS, motor starts via BEMF. If it doesn't start: check 0xB6 (bad_phase_count) for invalid commutation states, and verify SYS_AFE_REG1 bit positions against the SNR8503M reference manual.
Don't enable Phase 8.4 until needed. Refactoring speed PID from #define to g_flash_cfg is 6-10 hours of careful work touching PID.c / SpeedRamp.c / Protect.c. Only do it when a real use case demands runtime PID tuning. Today's bipropellant hosts can read+write the gain codes (0x85-0x87) and the values persist correctly to Flash — they just don't affect the running PID. Useful state-tracking even without affecting motor.
Consider a "release prep" mini-phase. Before any production deployment: (a) clean up Phase NN tags into semantic comments, (b) add CI for all 7 profiles, (c) write smoke-test host script that exercises every registered code, (d) bump FLASH_CONFIG_VERSION if any schema change, (e) update README's known-issues section. ~4 hours.

Closing thoughts

The project transformed a vendor-locked single-purpose firmware into a modular, multi-profile platform without losing the existing Hall-mode behaviour. The most valuable parts of the work were not the most complex parts — they were:

  • Reading the linker map carefully. One lks32mc03x_nvr.o string told us the vendor identity, license posture, and SDK ecosystem.
  • Using objcopy to rename closed-lib symbols. One command sidestepped what would have been a 10-15 hour reverse-engineering project.
  • Asking the code-review agent for a second opinion. The startup ASM bug would have shipped silently corrupting .data across every GCC flash. A 5-minute external review caught it.
  • Iterative fix cycles. 25 → 13 → 0 HIGH-severity findings showed the methodology converges.

The riskiest part of the work — sensorless commutation — is structurally complete but unverified on hardware. The board test plan exists; the firmware is ready. The remaining work is operational, not architectural.

Epilogue — GPL conversion (2026-05-13)

After the original retrospective, the project went one more step further: removing the proprietary closed-source .lib dependencies entirely so the final binary contains zero proprietary object code. This was done in several passes:

  1. First pass: write replacements (src/vendor_replacements.c). Started by decompiling the three vendor libs and writing function-by-function clean-room replacements. The most decomp-leaky function — Read_Trim, which contained the magic NVR-unlock constants 0x59000000 and 0x95000000 — was the highest IP-risk item.
  2. Second pass: eliminate Read_Trim. Realised its only caller (snr8503x_dac.c, for DAC calibration) could be patched to use identity defaults (DAC_AMC = 512, DAC_DC = 0) instead of reading NVR. Removed Read_Trim from vendor_replacements.c entirely. The most IP-risky piece — the NVR unlock magic — is now gone from the codebase.
  3. Third pass: discover most replacements were dead code. A code audit showed that BLDC_init, BLDC_init_snls, and 14 of the 15 globals (BLDC_1, BLDC_BUFF, BLDC_BUFF1, ICP_MODE, ICP_SetFLAG, and 7 other ICP_*CNT* variables) were never read by anything. They existed only because the original closed libs exported them; the firmware itself never consumed them. Removed them all.
  4. Fourth pass: inline what little remained. HALL_Update was a 1-line passthrough — inlined at its single caller. ICP_Loader was empty — removed its call site. The two real-work functions (COMP_Motor_Zero_Detect, Freerun_Zero_Detect) moved into src/COMP_Sensoeless.c where the rest of the sensorless logic lives. vendor_replacements.c deleted entirely.

The lesson

The journey from "linked closed libs" → "clean-room file alongside closed libs" → "no closed libs, dedicated replacement file" → "no closed libs, no replacement file, just inlined open-source code" was a useful exercise in reverse-engineering minimization. Each pass asked the same question: is this code actually needed, or are we just providing it because the original lib did?

For the SNR8503M / LKS32MC03x:

  • ~85% of the closed-lib code was license-gate machinery (NVR reads, equality checks, infinite-loop traps). Removing the calls leaves the firmware fully functional.
  • ~10% was dead state (variables that the lib wrote but nothing else ever read).
  • ~5% was real work (the two BEMF zero-crossing helpers) — and even those are textbook patterns from TI / ST / Microchip application notes.

The takeaway for similar small motor-controller chips: before assuming a closed lib does anything important, audit who calls it and read what they do with the results. Most closed motor-control libs at this price point are 90% chip-authentication and 10% commodity algorithm.
